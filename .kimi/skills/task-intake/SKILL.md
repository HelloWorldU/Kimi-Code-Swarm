# Task Intake — 任务接入门控

> 接到任何涉及修改代码 / 文件 / 配置的任务时，**最先执行本 Skill**。
> 强制「需求澄清 → 文档清单 → 场景与预期 → 用户对齐批准 → 才允许写实现」。

## 为什么

第一性原理：把「模式补全器」变成「可控的干活员」。

- 模式补全 = 任务看起来像 X → 直接输出 X 类方案通常的样子（只写 happy path、想当然加护栏、mock 凑数）。
- 工程 = 面对**这个**系统，对着事实验证，只做被要求的事。

本 Skill 强制后者。

## 铁律

**未经用户明确批准，禁止写任何实现代码。** 接到任务先走阶段 1–3，这三个阶段的产出是**文字**，不是代码。

## 阶段 1 — 需求澄清 + 文档清单

1. 用自己的话**复述需求**，列出你的**假设**。有歧义立即提问，不替用户做决定。
2. 列出**文档 / 文件清单**：本任务要读什么 —— 先看 `AGENTS.md` 快速决策树按图索骥，列出相关 `docs/`、`.kimi/skills/`、以及会被改动的源文件。
3. 实际读完清单里的内容，再进入阶段 2。

## 阶段 2 — 场景与预期结果

把任务拆成**具体场景**，每个场景写明**预期结果**。必须覆盖：

- ✅ **正常路径**（happy path）
- ⚠️ **失败 / 边界路径** —— 异常输入、空值、网络失败、并发、超长输入、多字节字符……
- 👤 **用户视角** —— 用户怎么操作、看到什么。涉及前端的任务，必须把交互（键盘、换行、折行、溢出）当成场景。

> 只写了 happy path = 阶段 2 没完成。

## 阶段 3 — 对齐门控（GATE）

把阶段 1 + 2 的产出**完整呈现给用户**，等待**明确批准**。

- 用户没有给出「可以 / 开始 / 通过」之类明确同意 → 不得进入阶段 4。
- 用户提出修改 → 回阶段 1 / 2 改完，**再次对齐**。

## 阶段 4 — 实现与验证

只有过了阶段 3 才动手写代码。期间遵守下列**不变量**。

> 用「不变量」而非「禁令」表述 —— 禁令能被换个写法绕过，不变量违反了能被当场检出。

- **真实现，不 mock** —— 做不了就停下来说明，不假装做完。
- **只做被要求的事** —— 不擅自加截断 / 上限 / 分页 / 护栏 / 重试等用户没要求的逻辑。
- **不动你没新增的东西** —— 觉得某段「没用」，先 grep 出它的**每一处引用**列给用户，确认后再删。
- **上下文完整** —— 凡涉及对话 / 历史，必须保住完整因果链；任何机制都不得丢弃或缩短早期信息。

验证：

- 跑**覆盖到本次改动的**校验，不是「主流程默认那个」。改了哪个包 / 模块，就跑那个包 / 模块自己的检查。
- 贴**原始输出**，不要只回一句「通过」。
- 删除前的引用 grep、改动后的校验，都要给用户看证据。

## 例外

纯问答、纯解释、用户明确说「直接改」的小改动 —— 可跳过阶段 1–3 的门控；但阶段 4 的不变量**永远生效**。
