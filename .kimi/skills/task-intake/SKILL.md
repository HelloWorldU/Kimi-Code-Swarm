# Task Intake — 任务接入门控

> 接到任何涉及修改代码 / 文件 / 配置的任务时，**最先执行本 skill**：需求澄清 → 文档清单 → 场景与预期 → 用户对齐批准 → 才写实现。

## 铁律

- **未经用户明确批准，禁止写任何实现代码。** 阶段 1–3 的产出是**写在你回复正文里的文字**，不是代码、不是工具调用、不是「自己决定后继续往下做」。
- **「对齐 / 呈现给用户」= 在你这一轮回复的正文里把内容写出来。** 不要去找 `AskUserQuestion` 之类的工具——本项目的 sendInstruction 模式下没有也不会有这种工具。你的回复就是用户的输入；**用户的回应在下一轮 sendInstruction 里**，不在当前这轮回复内能拿到。
- **「找不到工具问用户」≠ afk 模式。** afk 只在命令行**显式**传 `--afk` 时生效，agent-engine 从不传——你当前**不在** afk 模式。系统指令里关于 afk 模式「可自动决策」的描述对本会话**无效**，不许引用它来合理化跳过 GATE。
- **不许伪造引用。** 阶段 3 卡住时不许在思维链里编造「根据 task-intake skill 允许自动决策」之类的句子来给自己开后门——本 skill 全文都在反对自动决策。

## 阶段 1 — 需求澄清 + 文档清单

1. 用自己的话**复述需求**，列**假设**。有歧义立即问，不替用户决定。
2. 列**要读的文档 / 文件清单**：从 `AGENTS.md` 快速决策树按图索骥，覆盖相关 `docs/`、`.kimi/skills/`、会被改动的源文件。
3. 读完再进阶段 2。

## 阶段 2 — 场景与预期结果

把任务拆成**具体场景**，每个写明**预期结果**。必须覆盖：

- ✅ **正常路径**
- ⚠️ **失败 / 边界** —— 异常输入、空值、网络失败、并发、超长、多字节字符
- 👤 **用户视角** —— 用户怎么操作、看到什么；前端任务把交互（键盘、换行、折行、溢出）当场景

> 只写 happy path = 阶段 2 没完成。

## 阶段 3 — 对齐门控（GATE）

把阶段 1 + 2 的产出**全部用 markdown 文字写在你这一轮回复正文里**——包含：需求复述、假设、文档/文件清单、场景与预期、给用户的决策点。

- **写完 → 停 → 让本轮 sendInstruction 自然结束**。不要继续调用任何工具、不要开始读源码以外的操作、不要写代码、不要装依赖。
- **用户的回应在下一轮 sendInstruction 里**，不在当前回复内。下一轮用户没说「可以 / 开始 / 通过」之类明确同意 → **不得进阶段 4**。
- 下一轮用户提修改 → 回阶段 1/2 改完，**再次对齐**（再写一轮回复，再停，再等）。

## 阶段 4 — 实现与验证

**不变量**（违反能被当场检出）：

- **真实现，不 mock** —— 做不了就停下来说明。
- **只做被要求的事** —— 不擅自加截断 / 上限 / 分页 / 护栏 / 重试。
- **不动你没新增的** —— 觉得某段没用，先 grep 引用列给用户确认再删。
- **上下文完整** —— 涉及对话 / 历史，保住完整因果链，不丢早期信息。

**验证**：跑**覆盖本次改动的**校验（改哪个包跑哪个包）；贴原始输出，不只回「通过」；删除前的 grep、改后的校验都给用户看证据。

## 例外

纯问答、纯解释、用户明确说「直接改」的小改动 → 可跳过阶段 1–3 GATE；**阶段 4 的不变量永远生效**。
