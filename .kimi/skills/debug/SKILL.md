# Debug — 反复未解时的调试协作

> 触发：你尝试修同一类问题 N 次未解，或人类反复反馈同一症状仍未消失。
> 此时**停止猜测式修改**，加调试信息 + 通过 prompt 跟人类协作收集运行时数据。

## 铁律

- **反复未解 = 信息不够，不是创意不够。** 停止「换个写法再试」「换个 API 试试」「再加一层 try/catch」之类的猜测式修改。
- **你无法直接读 Tauri 控制台 / engine-state.json / OS 进程状态 / Kimi 内置日志**——需要这些数据时必须**显式让人类帮采**。不要假装看到了。不要基于猜测继续推进。

## 触发判据（满足任一即进本 skill）

- 同一症状人类反馈 ≥ 2 次仍未消失
- 你提出的修复假设 ≥ 2 个不同方向都被验证失败
- 你无法用现有代码 / 文档证据解释为什么旧行为会发生

## 调试协作步骤

1. **加日志**：在可疑路径用 `src/utils/logger.ts`（前端）或 `console.error`（agent-engine stderr）加 `log.info` / `log.error`，按 `docs/ARCHITECTURE.md` 的「日志分流」约定——stdout JSON 进 UI，stderr 进终端 / Debug Console。
2. **明确告诉人类要看什么 + 怎么采**：哪个文件改了；要不要重新打包；在哪里复现；要在 DEBUG Console 找什么前缀（如 `[Kimi]` `[Agent]` `[Git]`）；要看 JSON 的哪个字段；输出粘回哪。**让人类拷贝粘贴即可，不要让他读懂代码再翻译**。
3. **拿到数据再下一步**：人类反馈日志后，**基于数据**做下一步假设。**未拿到反馈前不允许再改代码**——这条违反 = 退回铁律第一条「停止猜测」。
4. **写 correlation key**：日志要带 `agent.id` / `kimiSessionId` / PR `#N` 之一，人类在并发输出里才筛得出相关行。

## 不变量

- **加完调试要交代怎么清**：修复落地后告诉人类哪些 log 是临时的、应删；哪些有长期价值、应保留（标 `// debug-keep:` 注释说明原因）。不留 `console.log` 残余。
- **commit message 要留根因**：不许「fix bug」「update files」之类敷衍。要写「为什么旧代码会错」+「怎么修对了」。
- **不要"绕过"重试**：人类没回反馈、自己又开始猜代码 = 直接违反铁律。停下，问。

## 退出

- 拿到数据 + 定位根因 + 修复 + 验证通过 → 回 task-intake 阶段 4 的不变量收尾。
- 多次协作仍无法定位 → 显式告诉人类「我已尽力，需要你手动介入调试」，停下；不要继续假装尝试。
